Modelo de Datos y Persistencia

Estrategias de Persistencia de Datos
Persistencia en Base de Datos
  • PostgreSQL 15: Sistema de gestión relacional robusto
  • Volúmenes Docker: Datos persistentes fuera del contenedor
  • ACID Transactions: Garantía de integridad de datos
  • Backups Automáticos: Respaldo programado de la BD
  • WAL (Write-Ahead Logging): Recuperación ante fallos
Volumen Persistente:
Los datos de PostgreSQL se almacenan en un volumen Docker (postgres-data) que persiste incluso si el contenedor se elimina o reinicia.
ORM - JPA/Hibernate
  • Mapeo Objeto-Relacional: Entidades Java ↔ Tablas SQL
  • Lazy Loading: Carga eficiente de relaciones
  • Caching L1/L2: Optimización de consultas repetidas
  • Transaction Management: Control automático de transacciones
  • Query Optimization: Hibernate genera SQL optimizado
Ventaja:
JPA abstrae la complejidad de SQL permitiendo trabajar con objetos Java mientras garantiza la persistencia en PostgreSQL.

Metodologías Ágiles: Scrum y Kanban

Adoptamos enfoques ágiles para mantener flexibilidad y adaptabilidad, valorando la comunicación efectiva y transparencia en todas las etapas del proyecto.

Scrum

Características:
  • Sprints iterativos (2-4 semanas): Trabajo dividido en iteraciones manejables
  • Roles definidos: PO, Scrum Master, Dev Team
  • Priorización: Tareas organizadas según valor de negocio

En este proyecto: Usamos Scrum para desarrollar el MVP inicial, organizando el trabajo en sprints de 2 semanas con daily standups y retrospectivas para mejora continua.

Kanban

Características:
  • Flujo continuo: Sin sprints, trabajo continuo
  • Tablero visual: To Do, In Progress, Done
  • WIP Limits: Límite de tareas en progreso
  • Pull System: Se toma trabajo cuando hay capacidad
  • Mejora continua: Optimización constante del flujo
Cuándo lo usamos:
  • Mantenimiento y soporte post-release
  • Corrección de bugs y hotfixes
  • Tareas de operaciones y DevOps
  • Trabajo con prioridades cambiantes
En este proyecto: Usamos Kanban para el mantenimiento post-release, gestionando bugs y mejoras continuas con un tablero visual que permite adaptación rápida.

Preguntas Técnicas Frecuentes

¿Cómo se comunica el Frontend con el Backend?
  • API REST: Comunicación mediante HTTP/HTTPS
  • JSON: Formato de intercambio de datos
  • Axios: Cliente HTTP en React para llamadas
  • CORS: Configurado para permitir peticiones cross-origin
  • JWT en Headers: Authorization: Bearer <token>
  • Status Codes: 200 (OK), 201 (Created), 400 (Bad Request), 401 (Unauthorized), 404 (Not Found)
¿Cómo funciona la autenticación JWT?
  1. Usuario envía email/password al endpoint /auth/login
  2. Backend valida credenciales con BCrypt
  3. Si es válido, genera un JWT firmado con clave secreta
  4. Frontend almacena el token en localStorage
  5. Cada petición incluye el token en header Authorization
  6. Backend valida el token en cada request protegido
  7. Si token es válido, permite acceso; si no, devuelve 401
¿Cómo se manejan las relaciones en la BD?
  • Foreign Keys: Restricciones a nivel de PostgreSQL
  • @ManyToOne: Muchas recetas → 1 país
  • @OneToMany: 1 receta → muchos ingredientes
  • @JoinColumn: Define la columna FK en JPA
  • Cascade: Operaciones en cascada (delete, update)
  • Fetch LAZY: Carga bajo demanda para optimizar
  • Bidireccional: Navegación desde ambos lados
¿Cómo se escala la aplicación?
  • Horizontal Scaling: Múltiples instancias del backend
  • Load Balancer: Nginx distribuye tráfico
  • Database Pooling: HikariCP gestiona conexiones
  • Caching: Redis para datos frecuentes
  • CDN: Contenido estático distribuido
  • Microservicios: Separar servicios por dominio
  • Auto-scaling: Kubernetes para orquestación
¿Cómo se manejan los errores?
  • @ControllerAdvice: Manejo global de excepciones
  • @ExceptionHandler: Captura excepciones específicas
  • ResponseEntity: Respuestas HTTP personalizadas
  • Try-Catch: Manejo local de errores críticos
  • Logging: SLF4J + Logback para trazabilidad
  • Frontend: Mensajes user-friendly con alerts
  • Validaciones: @Valid, @NotNull, @Size en DTOs
¿Por qué 13 tablas y no menos?
  • Normalización 3NF: Evita redundancia de datos
  • Separación de concerns: Cada tabla tiene responsabilidad única
  • Escalabilidad: Fácil agregar campos sin afectar otras tablas
  • Integridad: Foreign Keys garantizan consistencia
  • Performance: Índices específicos por tabla
  • Flexibilidad: Modificar una entidad sin impactar otras
  • Mantenibilidad: Código más limpio y organizado
¿Qué pasa si cae el servidor?
  • Docker Restart Policy: Reinicio automático de contenedores
  • Health Checks: Monitoreo continuo con /actuator/health
  • Persistence: Datos en volúmenes Docker, no se pierden
  • Backups: Respaldos automáticos de PostgreSQL
  • Logging: Logs persistentes para debugging
  • Alertas: Notificaciones de caídas
  • Failover: Réplicas de BD para alta disponibilidad
¿Por qué arquitectura REST y no GraphQL?
  • Simplicidad: REST es más simple de implementar
  • Estándar HTTP: Usa métodos nativos (GET, POST, PUT, DELETE)
  • Caching: HTTP caching funciona out-of-the-box
  • Documentación: Swagger/OpenAPI bien establecido
  • Stateless: Cada request es independiente
  • Compatibilidad: Funciona con cualquier cliente HTTP
  • Madurez: Más herramientas y documentación disponible
¿El sistema es persistente?

¿Los datos sobreviven reinicios y recreación de contenedores?

SÍ, completamente persistente
  • Volumen nombrado: postgres_data:/var/lib/postgresql/data
  • Declarado en: Configuración de Docker
  • Sobrevive a: Reinicios y recreaciones de contenedores
  • Se pierde solo si: Eliminas explícitamente el volumen
# Comandos que NO eliminan datos:
docker stop postgres_container
docker start postgres_container

# Elimina datos (¡cuidado!):
docker volume rm postgres_data
¿Cómo y cuándo se hacen los respaldos?

Backups automáticos configurados con systemd

  • Servicio: recetas-backup.service
  • Timer: recetas-backup.timer
  • Script: backup_full.sh
  • Frecuencia: Diariamente a las 02:00
  • Ubicación: /home/admin/api-recetas/backups
Rotación de Backups:
  • Script: backup_rotate.sh
  • Hora: 02:30 (después del backup)
  • Retención: Elimina archivos mayores a 14 días
¿Cómo restaurar desde un respaldo?

Pasos para restauración desde backup:

# 1. Copiar backup al contenedor
docker cp /home/admin/api-recetas/backups/init.sql \
  postgres_container:/tmp/restore.sql

# 2. Conectar al contenedor
docker exec -it postgres_container bash

# 3. Restaurar la base de datos
psql -U postgres -d recetas_db -f /tmp/restore.sql

# 4. Verificar restauración
psql -U postgres -d recetas_db -c "SELECT COUNT(*) FROM receta;"
⚠️ Importante: Ejecutar pg_dump desde dentro del contenedor para preservar UTF-8 correctamente.
¿El backup preserva UTF-8 y caracteres especiales?

¿Se pierden tildes/ñ al exportar?

❌ Problema conocido: Redireccionamiento desde PowerShell corrompe UTF-8
✅ Solución implementada:
  • Ejecutar pg_dump dentro del contenedor
  • Guardar en /tmp del contenedor
  • Usar docker cp para extraer el archivo
  • Verificación: init.sql regenerado con UTF-8 correcto
# Método correcto:
docker exec postgres_container \
  pg_dump -U postgres recetas_db > /tmp/backup.sql
docker cp postgres_container:/tmp/backup.sql .
