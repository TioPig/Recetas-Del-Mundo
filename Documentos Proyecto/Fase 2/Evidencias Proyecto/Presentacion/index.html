Resumen Ejecutivo

Actualmente somos dos los desarrolladores detrás de Recetas Del Mundo. Estamos transformando el arte culinario evolucionando desde un sistema monolito en Node.js y MySQL hacia una multiplataforma de tipo Docker y escalable en el tiempo. Nuestro propósito es rescatar recetas ancestrales, de madres, abuelas, etc. conectándolas con restaurantes, chefs y sus comensales. Para lograrlo, ofrecemos nuevas funciones modernas y amigables e interactivas en las cuales podemos destacar, valoración por estrellas, me gusta, favoritos, comentarios y donaciones, permiten entregar un valor adicional, y compartir las recetas. Así brindamos un servicio profesional, estandarizado y amigable que protege y potencia el patrimonio culinario a nivel global.

Estrategias de Persistencia de Datos
  • PostgreSQL 15 como motor relacional principal, con datos persistidos en volúmenes Docker.
  • Modelo normalizado en 13 tablas, con claves foráneas y restricciones para integridad referencial.
  • JPA/Hibernate para mapear entidades Java a tablas SQL y manejar transacciones.
  • Scripts SQL versionados para creación, migraciones y cargas iniciales de datos.
  • Backups periódicos de la base de datos para recuperación ante fallos.
Esquema de Base de Datos - 13 Tablas
Entidades Principales
  • usuario - Usuarios del sistema
  • receta - Recetas culinarias
  • ingrediente - Ingredientes por receta
  • categoria - Tipos de recetas
  • pais - Origen geográfico

  • comentario - Comentarios de usuarios

  • receta_del_dia - Destacados diarios
  • donacion - Contribuciones

Características del Modelo de Datos:
  • Normalización: Tercera forma normal (3NF)
  • Integridad Referencial: Foreign Keys en todas las relaciones
  • Índices: Optimizados para búsquedas frecuentes
  • Soft Delete: Campo "estado" para auditoría
  • Timestamps: fecha_creacion en todas las tablas
  • Validación: Estructura validada contra PostgreSQL 15

Seguridad del Sistema

  • Autenticación con JWT y expiración configurable, enviada en el header Authorization: Bearer <token>.
  • Control de acceso basado en roles (RBAC) con perfiles USER, ADMIN y LIDER.
  • BCrypt para hash de contraseñas y uso de variables de entorno para credenciales sensibles.
  • CORS y validaciones de entrada para prevenir SQL Injection y XSS.
  • Manejadores de errores globales en Spring Boot para respuestas HTTP consistentes.

Preguntas Técnicas Frecuentes

¿Cómo se comunica el Frontend con el Backend?
  • API REST + JSON sobre HTTP/HTTPS.
  • Axios como cliente HTTP en React.
  • CORS configurado y uso de Authorization: Bearer <token> para JWT.
¿Por qué arquitectura REST?
  • Simplicidad y compatibilidad con cualquier cliente HTTP.
  • Uso de estándar HTTP (GET, POST, PUT, DELETE) y caché nativo.
  • Documentación mediante Swagger/OpenAPI ampliamente soportada.
¿Por qué 13 tablas?
  • Normalización a 3NF para evitar redundancias.
  • Separación de responsabilidades por entidad (usuarios, recetas, valoraciones, etc.).
  • Integridad y rendimiento gracias a claves foráneas e índices específicos.
¿Qué pasa si cae el servidor?
  • Contenedores Docker con políticas de reinicio automático.
  • Datos persistentes en volúmenes y backups periódicos de PostgreSQL.
  • Logging para diagnóstico rápido y recuperación.
